**Conclusion:** None of the three DMS vendors expose a clean REST API to third-party developers. All three are partner-gated, and the realistic ingestion shape is **a scheduled file drop** (CSV/XML over SFTP) — i.e. the same data we already accept via the v0.1 CSV uploader, just delivered automatically. Most of T0.8 is **partnership procurement**, not engineering.

## 1. Per-DMS integration model

### 1.1 Frazer Computing

**Vendor URL:** https://www.frazer.com/integrations
**Help/docs:**
- `frazerhelp.com/help-manual/vehicleuploadoptions.htm`
- `frazerhelp.com/help-manual/vehicle_uploads.htm`
- `frazerhelp.com/help-manual/technical-details-and-custom-u.htm`

**Architecture:**
- Frazer is **desktop / on-prem** software (Windows). The dealer's machine pushes outbound data; there is **no Frazer-hosted cloud endpoint we could pull from**.
- The dealer configures Vehicle > 8 - Vehicle Uploads in Frazer, enters vendor credentials, picks a destination, and Frazer **takes a snapshot, converts inventory to `.csv` or `.txt`, and pushes it via FTP/SFTP** to the vendor server.
- Schedule: dealer-set daily time. Runs daily whether Frazer UI is open or closed *as long as the host PC is on*. (Single dealer downtime = upload stops.)
- Photos are included in the upload.

**API?** No partner REST API surfaced. Real-time partners (e.g. Dealer Car Search, Selly CRM) describe the integration as "API" in marketing, but Frazer's own docs only describe the FTP push mechanism. Treat "real-time API" as marketing language for "frequent FTP push" until founder verification.

**Partner program:** `frazer.com/frazer-partner-program` — application required (page was unreachable from our tooling; founder must confirm: cost, NDA, certification timeline, support SLA).

**Investigation status:** Partial. Mechanism = FTP/SFTP push **(verified by vendor docs)**. Partner program terms = **needs founder verification**.

### 1.2 DealerCenter

**Vendor URL:** https://www.dealercenter.com/partners/
**Support:** https://support.dealercenter.net/hc/en-us
**Contact:** 888-669-2669 (the only technical-integration-details channel disclosed publicly)

**Architecture:**
- DealerCenter is a **cloud DMS** (browser-based), so unlike Frazer they *could* expose a REST API. They do not publicly.
- The partner page lists 100+ inventory destinations (AutoTrader, Cars.com, CarGurus, Edmunds, etc.) — strong evidence DC produces a syndication-style outbound feed in a standard auto-marketplace format (XML or tab-delimited, similar to vAuto / HomeNet conventions).
- For **inbound** (DMS → DC) imports, support docs describe the partner DMS being notified to "establish an import feed" — i.e. SFTP-based handshake, partner-mediated.

**API?** Per public surface: no documented partner API. All integration appears to flow through an **account-management negotiation** with DealerCenter's partner team. The forum thread at `forum.dealerrefresh.com/threads/3775` independently confirms that fetching live inventory from a DMS via an open API "by region" is not generally available.

**Partner program:** Submit through `dealercenter.com/partners/`. Terms not public.

**Investigation status:** Mechanism = **unknown but most likely SFTP-delivered XML/CSV inventory feed**, gated by partner enrollment. Confirm via support call.

### 1.3 AutoManager (DeskManager / WebManager / DealerClick line)

**Vendor URL:** https://www.automanager.com/
**Product line:** DeskManager (DMS, desktop), DeskManager Online (cloud), WebManager (dealer website), AutomotiveCRM.
**Support:** https://support.automanager.com/

**Architecture:**
- DeskManager → WebManager is **internal-only** synchronisation; the public-facing docs say "automatic whenever changes are made" with no third-party hook.
- The only confirmed third-party data interface in their support knowledge base is **AGORA Data** (an external lender), which works by AutoManager **accepting** an AGORA-issued API key. This is inbound to AutoManager and tells us nothing about outbound access for us.
- Multiple independent sources (GetApp listing, dealer-software comparison sites) state plainly: **"AutoManager does not have an API available."**

**API?** No. Verified across vendor and third-party sources.

**Partner program:** None publicly advertised. Engagement appears to be ad-hoc through sales/support.

**Realistic ingestion paths, in descending preference:**
1. **Dealer-driven scheduled CSV/XLS export.** DeskManager has a built-in "export to spreadsheet" function. Dealer manually runs it (or schedules via OS task scheduler) → drops file into our SFTP / a watched Dropbox folder → we reuse the v0.1 CSV ingest path.
2. **Negotiated custom feed** with AutoManager support — likely requires a paid integration engagement.
3. **Browser-extension scrape of WebManager** — possible but ToS-hostile, brittle, and creates a marketplace-style "we became a bot" risk identical to the one we wrote ADR-001 to avoid. **Out of scope for v1.**

**Investigation status:** Mechanism = **manual export → SFTP** is the only path that doesn't require vendor cooperation. Founder verification needed if a higher-fidelity option exists.

## 3. Export format vs. our `vehicles` table

Our schema (verified from `supabase/migrations/0002_v01_core.sql:46`):

```
vehicles(
  id, dealer_id, stock_number (required, unique with dealer_id),
  vin, year, make, model, trim, mileage, price_cents, photo_url,
  description, status [available|pending|sold|hidden],
  created_at, updated_at
)
```

Existing CSV normaliser (`src/app/dashboard/inventory/actions.ts`) already handles:
- `priceToCents()` — strips `$` and commas, dual-handles `19999` vs `19999.00`
- `intOrNull(year, mileage)` — bounded ints
- `strOrNull(...)` — length-capped text
- Per-row error reporting with `{ row, reason }`
- Chunked upsert on `(dealer_id, stock_number)` — **same key works for DMS feeds**
- 5MB / 5000-row cap (we'd raise this only after confirming volume)

### 3.1 Typical DMS field set (synthesised from vendor docs + industry feeds)

| Field | Frazer CSV | DealerCenter feed | AutoManager export | Our column | Notes |
|---|---|---|---|---|---|
| Stock # | ✓ | ✓ | ✓ | `stock_number` | required, unique key |
| VIN | ✓ | ✓ | ✓ | `vin` | 11-17 chars; validate |
| Year | ✓ | ✓ | ✓ | `year` | bounded 1950-2100 |
| Make | ✓ | ✓ | ✓ | `make` | text ≤60 |
| Model | ✓ | ✓ | ✓ | `model` | text ≤80 |
| Trim | ✓ | ✓ | ✓ | `trim` | text ≤80 |
| Mileage / Odometer | ✓ | ✓ | ✓ | `mileage` | bounded |
| Asking / Internet price | ✓ | ✓ | ✓ | `price_cents` | reuse `priceToCents` |
| Cost / Book / MSRP | ✓ | ✓ | ✓ | — | **DROP at boundary** (privacy) |
| Photo URL(s) | ✓ (1-N) | ✓ (1-N) | ✓ (1-N) | `photo_url` | we store only the first; future: photos table |
| Description / comments | ✓ | ✓ | ✓ | `description` | text ≤4000 |
| Status / "for sale" flag | ✓ | ✓ | ✓ | `status` | map to enum |
| Body / Trim / Drivetrain / Engine / Transmission / Color / Doors | ✓ | ✓ | ✓ | — | **out of scope v1**; consider `vehicles_extra` later |
| Lot location / multi-rooftop ID | sometimes | ✓ | sometimes | — | **gap** — see §6 multi-rooftop risk |
| Date in inventory / age | ✓ | ✓ | ✓ | (derive from `created_at`) | useful for T2.3 auto-repost |
| Dealer-internal lender / floorplan / cost-of-funds | ✓ | ✓ | ✓ | — | **DROP at boundary** (PII / commercially sensitive) |

### 3.2 Mapping headlines

- **Common 11-column core is achievable** for all three vendors, mapping cleanly to our existing `vehicles` row.
- **Photos**: all three vendors emit multiple photo URLs. v1 keeps only the primary in `photo_url`. v2 should add `vehicle_photos(vehicle_id, url, sort_order)`.
- **Status mapping**: vendors don't use our four-state enum verbatim; we'll need a per-vendor mapper (e.g. Frazer "Sold but not delivered" → `pending`; "Wholesale" → `hidden`).
- **Schema drift**: vendors frequently add columns. The CSV parser tolerates this by ignoring unknown headers — DMS adapter should do the same.

## 4. Frequency

| DMS | Realistic ingestion cadence | Why |
|---|---|---|
| Frazer | **Daily batch at dealer-chosen time** | Vendor docs explicitly describe daily-scheduled FTP push; no webhook layer. Some partners advertise "real-time" but this appears to be tighter polling, not a vendor-emitted event. |
| DealerCenter | **Daily batch** initially, **possibly hourly** with partnership | Marketplace feeds from DC to AutoTrader/Cars.com run on 1-6 hour cycles in industry practice; we'd inherit whichever cadence the partner team configures. |
| AutoManager | **Daily batch (manual or scheduled)** | No event surface. Best the dealer can do without vendor cooperation is schedule a nightly export. |

**Implication:** we will not have sub-minute fresh inventory for T0.8 in v1. Buyer-facing chat (T0.2) already uses RAG over `vehicles`; staleness window of up to 24h is acceptable because:
- the vehicle row's `status` is the load-bearing field for "is this car still here" and dealers update that *after* the conversation
- conversations that hinge on minute-fresh state (auctions, just-sold) are <1% of LotPilot's traffic per founder's own segmentation

If a dealer hand-types a "just sold" message we should still write to `vehicles.status='sold'` directly from the inbox UI (already supported) rather than wait on the next ingest.

## 5. Mapping plan — proposed module shape

Reuse the existing pipeline. **Do not write code in this scope doc**; what follows is the *intended shape* once T0.8 is unblocked.

```
src/lib/dms/
  index.ts                — registry: { frazer, dealercenter, automanager } → DmsAdapter
  types.ts                — DmsAdapter, NormalisedVehicleRow, IngestResult (mirrors UploadResult)
  frazer/
    ingest.ts             — parse Frazer .csv/.txt, map columns, call shared normaliser
    schema.ts             — Frazer column → canonical column map
  dealercenter/
    ingest.ts             — parse DC outbound XML (likely vAuto-ish format)
    schema.ts
  automanager/
    ingest.ts             — parse DeskManager spreadsheet export
    schema.ts
  shared/
    normaliser.ts         — extracted from src/app/dashboard/inventory/actions.ts:
                             priceToCents, intOrNull, strOrNull, status enum mapper
    upsert.ts             — extracted: chunked upsert on (dealer_id, stock_number) +
                             returns {inserted, updated, errors[]}

src/app/api/dms/[vendor]/inbound/route.ts
                          — single inbound HTTP entrypoint
                          — auth: per-dealer HMAC (mirror marketplace pattern: deriveDealerSecret +
                             extension_secret_version column, already exists for v0.7.1)
                          — vendor param routes to adapter
                          — body = the raw feed file uploaded via webhook OR an SFTP-drop pointer

scripts/dms-sftp-bridge/  — optional sidecar; watches per-dealer SFTP directories and POSTs
                             new files into /api/dms/.../inbound. Not Next.js. v1 can defer this
                             by giving each dealer a Cloudflare R2 / S3 presigned PUT and treating
                             the resulting object-create event as the trigger.
```

**Refactor first, ingest second.** Step 1 of any T0.8 DMS work is to extract `priceToCents` / `intOrNull` / `strOrNull` / the upsert loop from `src/app/dashboard/inventory/actions.ts` into `src/lib/dms/shared/normaliser.ts` and have the CSV uploader call it. That way every adapter shares one battle-tested validator. Net delta to existing CSV route: zero behavioural change.

**Security:**
- Inbound endpoint MUST require `x-lotpilot-dealer-id` + per-dealer HMAC over the body (re-use the marketplace webhook pattern at `src/app/api/marketplace/inbound/route.ts`, including the versioned secret derivation shipped in v0.7.1).
- Inbound endpoint MUST validate file size BEFORE parsing (mirror the 5MB CSV check).
- Inbound endpoint MUST NOT trust `dealer_id` in the file body — `dealer_id` comes only from the authenticated header.
- Logger MUST mask VINs to last 6 (consistent with `system_warnings` phone-masking precedent).

## 6. Risks

1. **No real APIs anywhere.** All three vendors keep ingestion partner-gated. This is a *procurement* problem more than an engineering problem. Engineering time post-partnership is small (M per vendor). Procurement time may be **weeks-to-months** per vendor.
2. **Multi-rooftop dealers.** Frazer's CSV doesn't carry a rooftop/lot ID by default; a dealer with three lots running one Frazer instance will mix all VINs into one feed. Our `vehicles` table assumes one `dealer_id` per row. **T2.8 (multi-location) is a hard prerequisite for any multi-rooftop dealer on Frazer.** v1 ships single-rooftop only; reject multi-rooftop dealers at onboarding.
3. **Schema drift.** Vendors silently add columns and occasionally rename. Adapters MUST tolerate unknown columns (existing CSV parser already does) and MUST log+ignore unmapped fields rather than fail the batch.
4. **Photo hot-link decay.** Vendor-hosted photo URLs sometimes break when the vehicle is marked sold. v1 should snapshot photos to our own storage (R2/S3) — but that doubles storage cost. Defer to T0.8.1.
5. **Frazer host-PC offline.** If the dealer's Frazer machine is powered off at the scheduled upload time, the upload silently doesn't run. We should expose a "last ingest at" timestamp on the inventory page so dealers self-diagnose; if `now() - last_ingest > 36h`, raise a `system_warnings` row (same pattern as the Calendly warning shipped in v0.6).
6. **ToS gotchas.**
   - DealerCenter and AutoManager ToS likely prohibit scraping; we are not scraping in v1.
   - Frazer's partner agreement may include a "no direct competition" clause — if LotPilot were ever to add DMS-ish features (forms printing, deal jackets) we'd be in breach. Roadmap anti-feature list already says "No full DMS replacement" — keep it that way.
7. **Rate limits.** Not a v1 concern for batch ingestion. If we move to vendor APIs later, expect <10 req/s per dealer.
8. **PII bleed.** Frazer's full export includes buyer SSN/DL on sold deals. Our adapter MUST allow-list only the inventory columns and MUST refuse a feed file that contains buyer-PII columns at all (fail closed, raise warning).
9. **Photo URLs with credentials.** Some DMS exports embed FTP creds in the photo URL (`ftp://user:pass@cdn...`). Adapter MUST strip credentials before persisting.
10. **"Custom" upload destinations in Frazer.** Frazer's help doc mentions the partner-vendor list but is silent on whether a dealer can configure a non-listed FTP target. If yes, we have a no-partnership escape hatch for v1 pilots. **Needs founder verification.**

## 7. Go / no-go checklist

Before any engineering ticket gets opened against T0.8 DMS work, the founder must produce:

- [ ] **Frazer Partner Program** application submitted; signed MSA / NDA on file; partner ID issued; documentation packet received (field reference, sample CSV, FTP target spec).
- [ ] **Frazer escape-hatch confirmation**: confirm with Frazer support whether a dealer can paste arbitrary FTP creds into the Vehicle Uploads screen, or whether the destination is locked to the partner dropdown. (If escape-hatch exists, we can pilot one dealer immediately, before formal partnership lands.)
- [ ] **DealerCenter Partners** application submitted via `dealercenter.com/partners/`; verbal or written confirmation of feed mechanism (XML/CSV, SFTP target ours or theirs, cadence); sample feed file in hand.
- [ ] **AutoManager** support engagement scoped: at minimum, confirm whether DeskManager supports OS-scheduled CSV export to a local folder (likely yes, per WebManager docs); at best, paid integration quote.
- [ ] **One pilot dealer per DMS** willing to be the first-ever ingest. Without a live dealer-side configuration session, we have no way to discover the field-naming quirks each install exposes.
- [ ] **Multi-rooftop policy decision**: v1 = reject; v2 = build T2.8 first.
- [ ] **SFTP infra decision**: do we run our own SFTP (e.g. AWS Transfer Family, ~$200/mo flat + per-GB), or rely on dealer-uploaded files via a presigned PUT? Cost vs friction tradeoff; **founder to choose**.
- [ ] **Storage decision for photos**: snapshot to R2 (defensible, costs $) vs hot-link vendor URLs (free, breaks).

Until at least four of these eight boxes are ticked, **do not start coding** the DMS adapter. The work is *predominantly* outside-the-codebase.
