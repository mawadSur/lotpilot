## 1. KBB — Kelley Blue Book Vehicle Values (IDWS 4.0)

### 1.1 Developer portal
- **URL:** `https://developer.kbb.com/` (loads as a SPA; B2B intake at
  `https://b2b.kbb.com/industry-solutions/info-driver-web-service-idws/`).
- **API name:** InfoDriver Web Service (IDWS) 4.0. Legacy IDWS SOAP API
  exists; new integrations should target REST 4.0.
- **Verify via partner@kbb.com** (or `b2b.kbb.com/contact/`) — community
  reports (DealerRefresh thread) that the public application form is
  often unanswered and a direct named contact is the practical route.
  Founder relationship matters here.

### 1.2 Onboarding gate
- **NDA:** yes (required to receive the full API spec — the public
  developer portal does NOT publish endpoint shapes).
- **Partner agreement:** "InfoDriver Web Service licence" — separate
  from the consumer kbb.com TOS.
- **Approval:** IDWS 4.0 is "in pilot phase with select customers"
  (per public references). Use-case review (LotPilot's case: surfacing
  trade-in mid/low/high inside an inbound-buyer chat) must be approved
  before pilot creds are issued.
- **Estimated calendar time:** 30–60 days. Founder should expect to do
  a 30-min product walkthrough call.

### 1.3 Sandbox
- **Availability:** the IDWS 4.0 pilot is itself the sandbox — no
  separate publicly-documented test environment. Pilot creds are
  rate-limited; production creds rotate in after sign-off.
- **Throttling:** verify via partner@kbb.com — not published.

### 1.4 Auth model
- REST (per `b2b.kbb.com/.../idws/` — "REST-compatible service").
- **Verify via partner@kbb.com:** the public portal references both
  Bearer tokens and OAuth 2.0 as possibilities; the production model
  for IDWS 4.0 has not been confirmed publicly.
- The existing scaffold (`src/lib/trade-in/kbb.ts`) already reads a
  single `KBB_API_KEY` env var via `readKbbApiKey()` and the
  `kbbConfigured` gate — that matches a Bearer/API-key model. If KBB
  comes back with OAuth client-credentials, env.ts will need a
  `KBB_CLIENT_ID` + `KBB_CLIENT_SECRET` pair (mirror the Manheim
  shape).

### 1.5 Endpoint(s) we need
- **Trade-in value** specifically — not Fair Purchase Price (retail),
  not Auction (wholesale). KBB exposes all three values under IDWS;
  we only need Trade-In.
- **Path placeholder:** `POST /v1/valuations/trade-in` (already
  stubbed in `kbb.ts:51` — actual path TBD on contract).
- **Version:** IDWS 4.0.

### 1.6 Request fields
| Field      | Required | Notes                                          |
| ---------- | -------- | ---------------------------------------------- |
| `vin`      | optional | 17-char ISO 3779 (route already validates)     |
| `year`     | yes      | 1980 ≤ year ≤ currentYear + 1                  |
| `make`     | yes      | normalised string                              |
| `model`    | yes      | normalised string                              |
| `mileage`  | yes      | int 0–500k                                     |
| `condition`| yes      | excellent/good/fair/poor (KBB's own taxonomy)  |
| `zip`      | yes      | 5-digit; KBB values are region-adjusted        |

Route handler already validates all of the above
(`src/app/api/dashboard/trade-in/value/route.ts:51-117`); the
adapter does not re-validate.

### 1.7 Response → `TradeInResult` mapping
KBB's IDWS Trade-In response returns a `valuesByCondition` block. Our
`TradeInResult` shape is:

```ts
{ available: true, provider: "kbb",
  low_usd:  number,  // KBB "below average" or condition-specific low
  mid_usd:  number,  // KBB "typical" / "fair purchase"
  high_usd: number }
```

- Map KBB "trade-in low" → `low_usd`, "trade-in fair" → `mid_usd`,
  "trade-in high" → `high_usd`. Confirm exact field names against the
  NDA'd spec; verify via partner@kbb.com.
- If KBB returns only a single value for the requested condition, mid
  takes that value and low/high get a ±10% sleeve (document the
  derivation in the adapter; do NOT silently invent numbers).

### 1.8 Rate limits + pricing tier
- **Verify via partner@kbb.com.** Not publicly posted. Industry
  rumour: per-call ($) with volume tiers + a monthly minimum. KBB
  Batch VIN (IDBV) has a separate range-based price.
- Assume worst case ~$0.05–0.25 per call for the first-pilot tier
  until contract lands. Budget at 1k calls/dealer/month × 50 dealers =
  50k/month → $2.5k–$12.5k/mo.

### 1.9 PII handling
- KBB does NOT need consumer PII (no SSN, no DOB, no name) — only
  vehicle attributes + zip. **This is the dealer's car-info call, not
  a consumer-data call.**
- Liability surface: minimal. Zip code is the only quasi-locator and
  it's already on the inbound message.

### 1.10 Switch-on plan
1. Set `KBB_API_KEY` in Vercel production env.
2. Set `TRADE_IN_PROVIDER=kbb` in Vercel production env.
3. Deploy. `kbbConfigured` flips to true at module load; dispatcher
   routes to `kbbProvider.value()`.
4. **Adapter work required:** replace the TODO throw at
   `src/lib/trade-in/kbb.ts:63` with the real fetch + response map
   (Section 1.7). Keep the AbortController + 5s timeout exactly as
   scaffolded.
5. Rotation: KBB keys rotate via partner-portal regen. Plan to rotate
   every 90 days; add `KBB_API_KEY_PREV` if KBB supports overlapping
   keys (mirror `MARKETPLACE_MASTER_SECRET_PREV` in v0.7.1).

## 2. Manheim MMR — wholesale valuations

### 2.1 Developer portal
- **URL:** `https://developer.manheim.com/`
- Documentation portal: pages for authentication, environments,
  valuations endpoint paths are publicly visible.
- Public note (verified): the legacy Mashery-backed Valuations API "is
  no longer available for onboarding" and new partners are routed to
  the **Cox Automotive API Storefront** at
  `https://developer.coxautoinc.com/`. Treat the Cox Auto storefront
  as the canonical onboarding surface; the legacy portal is reference
  material only.

### 2.2 Onboarding gate
- **NDA / Partner agreement:** Manheim runs a "careful review and
  evaluation" before granting access. Applicants must specify:
  - which APIs they intend to use
  - their consumption approach
  - whether they're a direct Manheim customer or a third party
    acting on behalf of a customer (we are the latter)
- **MMR Additional Terms** apply on top of the master agreement —
  these govern display, redistribution, and dealer-of-record
  constraints. Verify via Cox Automotive legal that LotPilot's
  "show MMR mid to a buyer mid-conversation" pattern is permitted
  under the third-party-platform clause. Worst case: MMR display may
  be limited to authenticated dealer dashboard surfaces and forbidden
  in consumer-facing chat.
- **Approval contact:** `DataSyndication@coxautoinc.com` (general),
  `api-support@manheim.com` (per-application registration).
- **Estimated calendar time:** 30–45 days for legal + Mashery
  registration.

### 2.3 Sandbox
- **Yes — three named environments:**
  - QA: `integration1.api.manheim.com`
  - UAT (pre-prod): `uat.api.manheim.com`
  - Prod: `api.manheim.com`
- "Access may be granted to each environment independently based on
  the API consumer's needs."
- Separate credentials per env (standard Mashery model). Throttling:
  "Manheim configures quotas for each API client to constrain the
  maximum number of requests per second and per day" — exact numbers
  not published; verify via api-support@manheim.com.

### 2.4 Auth model
- **OAuth 2.0**. Two grant types supported: Client Credentials and
  Resource Owner Password Credentials (RFC 6749). LotPilot wants
  Client Credentials — we don't impersonate dealer users.
- Token endpoints:
  - UAT: `https://uat.api.manheim.com/oauth2/token.oauth2`
  - Prod: `https://api.manheim.com/oauth2/token.oauth2`
- Credentials are a Mashery "package key" + "secret" → used as the
  OAuth2 client_id + client_secret.
- **Important:** Manheim explicitly tells clients to cache the access
  token until expiry and refresh on 401 — DO NOT poll-refresh on a
  timer.

### 2.5 Endpoint(s) we need
- **Valuations API**, VIN-driven endpoint:
  - `GET /valuations/vin/{VIN}`
  - `GET /valuations/vin/{VIN}/{SUBSERIES}`
  - `GET /valuations/vin/{VIN}/{SUBSERIES}/{TRANSMISSION}`
- Optional query params: `odometer`, `region`, `zipCode`, `color`,
  `grade`, `include=ci` (confidence interval), `include=retail`,
  `date` (back to 2018-11-01).

### 2.6 Request fields
Only the VIN is mandatory in the URL. Our `TradeInPayload` carries
year/make/model + mileage/zip — Manheim derives YMM from the VIN, and
`odometer=mileage` + `zipCode=zip` go on the query string.

If VIN is absent (caller passes only YMM), use the YMMT search
endpoint (`/valuations/ymmt/...`) — but most of our trade-in callers
will have a VIN.

### 2.7 Response → `TradeInResult` mapping
Manheim returns:
- `wholesale.average`, `wholesale.above`, `wholesale.below`
- `adjustedPricing.wholesale.{average,above,below}` (after mileage
  + condition adjustments)
- `retail.{average,above,below}` (when `include=retail`)
- `confidenceInterval.priceRange.{adjustedHigh,adjustedLow}` (when
  `include=ci`)
- `averageOdometer`, `averageGrade`, `sampleSize`, `currency`

Map (prefer adjusted over base — adjusted reflects mileage):

```ts
{ available: true, provider: "manheim",
  low_usd:  adjustedPricing.wholesale.below,
  mid_usd:  adjustedPricing.wholesale.average,
  high_usd: adjustedPricing.wholesale.above }
```

Drop the result with `available: false, reason: "low_confidence"` when
`sampleSize < N` (verify N with Manheim; suggested floor: 20).

### 2.8 Rate limits + pricing
- **Pricing:** verify via DataSyndication@coxautoinc.com. Industry
  reputation: tiered with a meaningful monthly minimum (Manheim was
  hit 2.6B times last year — they price for institutional volume,
  not independent-dealer-by-dealer).
- **Rate limits:** per-package, per-second + per-day — exact numbers
  verify via api-support@manheim.com.

### 2.9 PII handling
- Same as KBB: no consumer PII. VIN + vehicle attributes only.
- Liability surface: data licence compliance — MMR redistribution is
  contractually restricted. **Verify with legal:** displaying MMR mid
  to a consumer in a chat reply may require explicit licence terms.
  Default-safe posture: surface MMR to the *dealer* in their dashboard
  and let the dealer paste/adapt into the buyer chat manually until
  the licence is confirmed.

### 2.10 Switch-on plan
1. Set `MANHEIM_CLIENT_ID` + `MANHEIM_CLIENT_SECRET` (Mashery package
   key/secret) in Vercel.
2. Set `TRADE_IN_PROVIDER=manheim` in Vercel. (Only flip if KBB is
   unavailable or coverage gap — KBB stays primary.)
3. Deploy. `manheimConfigured` flips true; dispatcher routes to
   `manheimProvider.value()`.
4. **Adapter work required:**
   - Token exchange step (OAuth2 client_credentials) with in-memory
     token cache keyed by env. **Cache the token until expiry minus
     30s; refresh on 401.**
   - GET `/valuations/vin/{VIN}` with `odometer` + `zipCode` + the
     `include=ci` query param.
   - Response map per Section 2.7.
   - Replace the TODO throw at `src/lib/trade-in/manheim.ts:49`.
5. Rotation: Mashery supports key regeneration; plan 90-day cycles.
   No overlap-window support documented — coordinate maintenance
   windows.

- Specific products we care about:
  - **Credit Application Field Service** — defines required fields

  - **On-Demand APIs** — the partner-microservices layer that powers

  - **Digital Retail Services prequalification** (note: this product

### 3.2 Onboarding gate
- **NDA + DSP/Integration Partner agreement.** RouteOne curates its
  partner ecosystem (200+ DSPs); they vet new integrations against
  the F&I workflow.
- **Approval:** technical + business review. They will want to know:
  - what data flows in/out
  - which finance sources we plan to submit to (RouteOne is a
    multi-lender hub — submission goes to lenders the dealer is
    enrolled with)
  - dealer-of-record posture
- **Estimated calendar time:** 30–60 days; certification step adds
  ~21 days.

### 3.3 Sandbox
- **Verify via partner@routeone.com.** RouteOne's UAT environment is
  not publicly described. Industry practice: separate UAT creds,
  separate dealer-test-IDs, certification required before prod
  flip-over.

### 3.4 Auth model
- **Verify via partner@routeone.com.** Not publicly published.
  RouteOne's "On-Demand APIs" sit behind a partner gateway —
  candidates are OAuth2 client-credentials and/or mTLS (financial-data
  partners commonly require client cert). Plan env shape to support
  whichever: a key alone (`ROUTE_ONE_API_KEY`) or a cert pair.
- The existing scaffold env (`ROUTE_ONE_API_KEY` + `ROUTE_ONE_DEALER_ID`)
  accommodates a Bearer-token model. If mTLS, we'll add
  `ROUTE_ONE_CLIENT_CERT_PEM` + `_KEY_PEM` and pass via the fetch
  agent.

### 3.5 Endpoint(s) we need
For T1.6 the minimum is **soft-pull / pre-qualification submission**,
not full credit application. Verify the exact endpoint with
RouteOne — candidates include:
- "Prequal" service (consumer pre-qual, returns score band + offer
  range without finance-source submission); RouteOne's prequal is
  powered by 700Credit's QuickQualify under the hood.
- "Credit Application Field Service" — overkill for T1.6 v1; defer
  until T2.x when we want full submission to a specific lender.

### 3.6 Request fields
Verify via partner@routeone.com. Soft-pull-style call should require:
- `first_name`, `last_name`, `dob`, `ssn_last4`, address, monthly
  income, requested amount — already on our `PrequalPayload`
  (`src/lib/financing/types.ts:13-21`).
- Plus: `dealer_id` (RouteOne's, NOT our internal one).

**HARD RULE (v0.7.1 enforced):** the full 9-digit SSN is rejected at
the route handler. If RouteOne's prequal endpoint requires a full SSN,
that route is **not eligible for our pipeline as-is** and we either
- pick the soft-pull-style endpoint that needs only ssn_last4 (700C
  QuickQualify style), or
- move full-SSN flows into a dealer-supervised hosted-form, never
  through our API.

### 3.7 Response → `PrequalResult` mapping

```ts
{ available: true, provider: "route_one",
  status: "approved" | "declined" | "pending",
  reference_hash: sha256(routeone_application_id) }
```

Verify response shape with RouteOne. They likely return an
`applicationId` + `decision` + a per-finance-source breakdown if we
go full-submission. For T1.6 v1 we collapse the multi-lender response
into the single status above.

### 3.8 Rate limits + pricing
- **Verify via partner@routeone.com.** RouteOne typically prices the
  DSP integration with a per-dealer-per-month fee + per-submission
  rate; pricing varies by volume tier and product mix.
- Soft-pull pricing (delegated to 700C underneath) is the 700C rate
  card — see §4.

### 3.9 PII handling — CRITICAL
- The pre-qual call carries the most-sensitive PII surface we touch.
  Our v0.7.1 hard rules (`src/app/api/dashboard/financing/prequal/route.ts:1-35`)
  must remain inviolate when the live adapter lands:
  1. Reject `\b\d{9}\b` in the raw body **before** parsing.
  2. Accept only `ssn_last4`, regex `/^\d{4}$/`.
  3. Log only the WHITELIST in `route.ts:201-209` (provider, status,
     request_id, reference_hash, available). No body. No
     JSON.stringify(payload). No error.message in response body.
  4. `reference_hash = sha256(provider_id)` — opaque-but-traceable.
- **Data retention:** RouteOne stores the application server-side
  (they are the dealer's credit-submission system of record). Our
  responsibility: never persist the request body locally, never
  echo it back, store only the `reference_hash`.
- **FCRA permissible purpose:** consumer-initiated pre-qual via a
  dealer's website establishes permissible purpose (per public
  FCRA-soft-pull guidance). Founder needs to confirm with RouteOne
  that LotPilot's chat-initiated pattern qualifies — if a buyer is
  asking "what would my payment be?" that's consumer-initiated, but
  the documentation of consent must survive an audit. **Likely
  requires an explicit consent step in the dashboard form (checkbox
  + timestamp logged).**
- **Liability surface:** RouteOne (and the underlying bureau resale
  partner) carries the bureau-data liability. LotPilot's liability:
  faithful transport, no logging, scope-of-consent.

### 3.10 Switch-on plan
1. Set `ROUTE_ONE_API_KEY` + `ROUTE_ONE_DEALER_ID` (and possibly cert
   env vars) in Vercel.
2. Set `FINANCING_PROVIDER=route_one`.
3. Deploy. `routeOneConfigured` flips true; dispatcher routes to
   `routeOneProvider.prequalify()`.
4. **Adapter work required:** replace the TODO throw at
   `src/lib/financing/route_one.ts:21`. Add token cache if OAuth, add
   mTLS agent if cert-bound. Map response per §3.7. Add an explicit
   consent-capture column on the prequal form (FCRA paperwork).
5. **Rate-limit tightening:** before any live call, tighten the
   prequal route's rate limit from the v0.7.1 reused `dealer` rule
   (120/60s) to the v0.7.2 plan: dedicated rule, 3/hour/dealer
   (file header at `route.ts:25-34` documents this). Single dealer
   should never burn 120 SSN-last4 attempts per minute against a
   live bureau.
6. Rotation: per RouteOne partner agreement. Plan 90-day rotations.

  700Credit's reseller agreement covers it.

  chat-driven soft-pull is acceptable. (Public materials list "live
  chat" as a supported integration point — so this should be a

### 4.3 Sandbox

### 4.5 Endpoint(s) we need

  own customer interface (us). 700Credit hosts no UI on our side; we

### 4.6 Request fields — **CRITICAL FINDING**
**QuickQualify soft pull requires only NAME + ADDRESS. No SSN. No DOB.**
(700Credit public materials: "Consumers fill out a short form with
name and address to get their results.")

This is a **major simplification** vs the current `PrequalPayload`
shape, which requires `dob`, `ssn_last4`, monthly income, and
requested amount. Two options:

1. **Slim payload for 700Credit path only:** the adapter ignores
   `dob`/`ssn_last4`/`monthly_income`/`requested_amount` and submits
   only name + address. The route still validates them so the
   dashboard UX is consistent across providers.
2. **Make those fields optional in `PrequalPayload`:** breaks the
   v0.7.1 contract. Avoid for now.

Option 1 is the right call for v0.7.2.

If 700Credit is the chosen path, the existing route handler's
SSN-last4 enforcement is belt-and-braces only — the data isn't sent
upstream. The whitelist log policy still applies.

### 4.7 Response → `PrequalResult` mapping
700Credit returns three formats: summary XML, full credit file XML,
trade summary data. We need:
- `FICO score` (number)
- `Vantage score` (number, where available)
- Decision band (approved/declined/pending equivalent — verify exact
  field names)

Map:

```ts
{ available: true, provider: "seven_hundred_credit",
  status: <decision>, // map 700C decision → our enum
  reference_hash: sha256(700C_reference_id) }
```

Score bands themselves don't go on the wire response — they belong
in the dashboard surface, not in the API envelope. Add an optional
`score_band: "prime" | "near_prime" | "subprime"` to `PrequalResult`
if the founder wants to render bands in the UI (separate v0.7.3
spec).

### 4.8 Rate limits + pricing
- **Verify via sales@700credit.com.** Not publicly posted. Industry
  convention for soft-pull pre-quals: ~$1–$3 per pull at low volume,
  with monthly-minimum tiers ($200–$500/mo for independent dealers).

### 4.9 PII handling
- Soft-pull = no SSN, no DOB. **Lowest PII liability surface of the
  four providers.**
- 700Credit handles the bureau call, stores the credit file, and
  delivers FICO + summary to the dealer. We store only
  `reference_hash` + decision + (optional) score band.
- FCRA permissible purpose: consumer-initiated soft-pull with
  on-screen consent.
- Liability: 700Credit + bureau carry the credit-data liability.
  LotPilot: transport, consent capture, no logging of PII.

## 6. Open questions for the founder (in priority order)

1. **Manheim MMR display-in-chat licensing.** The MMR Additional Terms
   may restrict surfacing wholesale values to a *consumer* (vs the
   dealer dashboard). If so, T1.5's "show trade-in mid in the buyer
   chat" pattern needs to be KBB-only and Manheim becomes a
   dealer-dashboard-only data source. Need legal confirmation before
   building.

2. **RouteOne vs 700Credit primacy.** RouteOne's pre-qual is *itself*
   a 700Credit integration. Paying for both is wasteful. Recommend:
   start with 700Credit direct (faster onboarding, no SSN needed,
   lower PII surface, $/pull vs RouteOne's per-dealer SaaS fee), and
   only add RouteOne later if dealers need full credit-app submission
   to specific lenders (a T2.x feature, not T1.6).

3. **FCRA consent-capture surface.** Whether RouteOne or 700Credit,
   the consumer's "yes, soft-pull my credit" needs an auditable
   timestamp + IP + scope. v0.7.1 scaffold has no such table — adding
   it is a prerequisite for any live financing call. Founder needs to
   decide: in-chat consent (legally riskier — buyer is the consumer,
   the chat is the surface) or always-dealer-supervised consent (the
   dealer enters the pre-qual on behalf of a buyer in person, with
   their wet-signed FCRA disclosure). The latter materially changes
   the UX from "AI runs pre-qual in chat" to "AI offers, dealer
   completes."
