## 1. Goal & non-goals

### 1.1 Goal

Generate vertical 9:16 short-form videos (Reels / TikTok / YouTube Shorts /
Marketplace video listings) **automatically from existing inventory rows**.
Each video should be:

- Built from the vehicle's existing `photo_url` (single hero image at
  v0.1 — see §1.3 / Open Question OQ-1), with Ken-Burns motion,
  layered captions, and a price/CTA card.
- Scripted by Claude reusing the same dealer-voice prompt machinery as
  T2.2 (the listing optimizer), so the spoken/written copy stays
  consistent with how the dealer already talks on Marketplace.
- Producible in **bulk** (a 30-vehicle lot should be batchable
  overnight) without a human in the editing loop.
- Cheap enough that a $99–$199/mo dealer plan absorbs 1 video per
  vehicle per month without margin pain (§8).

The strategic point: independent used-car dealers on Marketplace + SMS
will never run a TikTok creative shop. Whoever **automates the
top-of-funnel social distribution** for them turns LotPilot from
"reply faster" into "drive more demand," which is the difference
between a $99 tool and a $499 tool.

### 1.2 Non-goals (v1)

- ❌ **In-browser video editing UI** with a timeline, drag handles,
  per-frame trim. We don't have UX bandwidth and Shotstack/Creatomate
  already ship hosted studios if a dealer wants one — we'll link out.
- ❌ **Royalty-free music library curation.** Bundling 200 licensed
  tracks is its own legal project. v1 ships **no background music**
  (silent + captions, or TTS voiceover only). See §9.3.
- ❌ **Direct publishing to TikTok / Reels.** Meta forbids server-side
  posting to a personal Marketplace account (see ADR for T0.9). We
  deliver a downloadable MP4 + caption text the dealer pastes
  manually, exactly like the relay flow.
- ❌ **Live streaming, multi-vehicle compilations, "best of the lot"
  reels.** Maybe Phase 3 (§10).
- ❌ **AI-generated b-roll or photos.** We ride on photos the dealer
  already uploaded. If a vehicle has no photo, we surface that
  pre-flight (§9.2).

### 1.3 Hard scope-creep guardrail

If a question starts with "could we also...," the answer in v1 is no.
The goal of v1 is **one button → one MP4 → one paste**. Everything
else is a v2 problem after we see retention data.

## 2. Output spec

| Property            | Value                                                    | Why                                                      |
| ------------------- | -------------------------------------------------------- | -------------------------------------------------------- |
| Resolution          | 1080 × 1920                                              | Universal 9:16 for Reels / TikTok / Shorts / FB Stories  |
| Container / codec   | MP4 / H.264 (yuv420p) + AAC audio                        | Widest device + Marketplace compatibility                |
| Frame rate          | 30 fps                                                   | TikTok recommends ≥30; 60 is unnecessary for stills      |
| Duration            | 15–30 s (template-locked, see §5.3)                      | TikTok/Reels sweet spot; ≤30s skips skip-rate cliff      |
| Bitrate target      | ~2.5 Mbps video / 128 kbps audio                         | Keeps file ≤8 MB at 30s                                  |
| File-size budget    | **≤ 8 MB**                                               | Below Meta/TikTok upload friction; mobile data-friendly  |
| Captions            | **Burned in** (not sidecar SRT)                          | TikTok/Reels autoplay muted; sidecars get stripped       |
| Caption font        | Bundled Inter SemiBold (SIL Open Font License)           | Avoid runtime font fetches (§9.2)                        |
| Color profile       | BT.709, full-range                                       | Default for social platforms                             |
| Audio track         | Phase 1: silent. Phase 2: TTS voiceover. (§10)           | Phased to control cost + IP risk                         |
| Aspect-safe zones   | Top 220 px + bottom 350 px reserved for UI/captions      | TikTok overlay + creator-handle bar                      |
| Suggested fallback  | 720 × 1280 if 1080p render exceeds budget                | Quality knob — set per-dealer or per-tier later          |

### 2.1 Visual template (v1, locked)

```
[0.0s–1.0s]  Black → fade-in hero photo with slow Ken-Burns zoom
[1.0s–4.0s]  Caption: "{year} {make} {model}"  + price tag (bottom)
[4.0s–10.0s] Photo pans + 2–3 spec captions (mileage, trim, features)
[10.0s–18.0s] Photo zooms back to fit + AI-written hook line
[18.0s–22.0s] Dealer name + signature + Calendly QR (if dealer has one)
[22.0s–25.0s] CTA card: "DM us / Tap for details" — fade to black
```

All timings, fonts, colors, motion paths come from a **single JSON
template** kept in `src/lib/video/templates/standard-9x16.v1.json`
(see §11 — DB schema also stores the template_id used for each render
so we can A/B later).

## 3. Pipeline options — comparison

Four candidates evaluated. All numbers verified May 2026 (see Sources at
end of doc); anything not verified is marked `verify`.

| Criterion                | FFmpeg on Vercel fns                          | **Remotion + Lambda**                          | Shotstack (hosted)                       | Creatomate (hosted)                  |
| ------------------------ | --------------------------------------------- | ---------------------------------------------- | ---------------------------------------- | ------------------------------------ |
| Render model             | We orchestrate ffmpeg in-process              | React-component scenes → AWS Lambda           | JSON template POST → hosted render        | JSON template POST → hosted render   |
| Cold-start latency       | ~1–3 s (fluid compute warm pool)              | ~1–4 s Lambda cold, ~200 ms warm               | Queue + ~5–15 s `verify`                 | Queue + ~5–10 s `verify`             |
| Render speed (30s 1080p) | 8–25 s (single-thread)                        | 4–10 s (parallel chunks)                       | 10–30 s wall time                        | 8–20 s wall time                     |
| Cost per render          | $0 infra + Vercel CPU-time only               | ~$0.001–$0.01 (AWS Lambda + S3)                | $0.20 / min → **~$0.10 per 30s**         | ~14 credits/min @720p ≈ $0.027–0.07  |
| Free tier                | n/a                                           | n/a (your AWS account)                         | 25-credit minimum buy                    | 50 credits free                      |
| Library size on disk     | ffmpeg static binary ~80 MB → Vercel layer    | Bundled in Lambda, ~200 MB cold image          | Zero (hosted)                            | Zero (hosted)                        |
| Dev complexity           | High (ffmpeg arg-string hell, font/codec QA)  | **Medium** — React + Remotion CLI + Lambda IAM | Low (REST + JSON)                        | Low (REST + JSON)                    |
| Customization ceiling    | Total — anything ffmpeg can do                | Total — anything React can render              | Medium (Shotstack EDL DSL)               | Medium (Creatomate template DSL)     |
| Vendor lock-in           | None                                          | Low (you own the React; Lambda is portable)    | High (proprietary EDL)                   | High (proprietary template format)   |
| License                  | LGPL/GPL ffmpeg — fine for SaaS dynamic link  | Source-available; **company license** required when revenue > threshold | Commercial SaaS                          | Commercial SaaS                      |
| Vercel-function fit      | Fits 300s default Fluid Compute timeout       | Async webhook from Lambda — clean              | Async polling or webhook                 | Async polling or webhook             |
| Operational surface      | We own ffmpeg flags, fonts, asset pipeline    | We own Lambda function + S3 + IAM              | Zero infra — just an API key             | Zero infra — just an API key         |
| Failure-domain blast     | Render failure = our bug                      | Lambda crashes = our bug; rate-limit risk      | Their outage = our outage                | Their outage = our outage            |

### 3.1 Recommendation: **Remotion + Lambda** (with FFmpeg as the
"break glass" alternative)

**Why Remotion wins for LotPilot specifically:**

1. **Cost dominates at scale.** At our cost envelope (§8), per-render
   cost is the biggest sensitivity. Lambda is 10–50× cheaper than
   Shotstack/Creatomate per render. Over 12 months × 100 dealers × 30
   vehicles, that's the difference between ~$36 and ~$1,800 in
   render fees alone.
2. **Template velocity matters.** The video template _will_ iterate
   weekly in the first 2 months. React JSX with TypeScript types is
   dramatically faster to change than an FFmpeg filtergraph
   string-soup or a vendor's proprietary EDL. Designers and the
   founder can read JSX; nobody reads `-vf "drawtext=fontfile=..."`.
3. **No vendor lock-in.** If Remotion's license changes (it has a
   commercial threshold), we can in-source the render with
   `@remotion/renderer` on our own container — same React code.
4. **Bilingual / dynamic content** is trivial. The script & captions
   come from Claude as a JSON object (§5); React composes that into
   the timeline declaratively. Doing the same in FFmpeg requires
   pre-generating PNGs and overlaying them.
5. **Lambda timeout is generous enough.** Even at the worst case
   (1080p 30s video with TTS-paced captions and 4-second renders) we
   stay well inside Lambda's 15-min limit — Vercel's 300s function
   ceiling is irrelevant here because Remotion renders happen
   _inside Lambda_, not in our Next.js function.

**Why not Shotstack/Creatomate as v1:**

- They are the right answer for someone shipping next week and
  willing to pay $0.10/render. We're shipping over weeks and need
  cost discipline at 100×30 scale.
- Reasonable Phase 0 contingency: if Remotion setup blocks for >3
  days, we ship Shotstack first and migrate. See §10.

**Why not raw FFmpeg as v1:**

- Maintenance tax. Every new template variation = a new flag
  permutation. The codebase already has enough surface area;
  ffmpeg-as-design-system is a tax we'll pay forever.
- The 80 MB binary on Vercel adds cold-start risk to a function path
  the dashboard depends on. We could side-car it on a different
  runtime, but at that point we're rebuilding Remotion Lambda
  ourselves with worse ergonomics.

**Break-glass alternative:** keep a thin FFmpeg renderer (Phase 0:
implemented as `src/lib/video/renderers/ffmpeg-fallback.ts`) for
local dev / Vercel-only mode, so a dealer demo doesn't require
working AWS creds. This is a debug-only path, not a production path.

> Note: ElevenLabs retail per-character pricing is messy — they
> sell credits per plan with Creator $0.30/1K, Pro $0.24/1K, Scale
> $0.18/1K overage. Numbers above are scaled to $/M. **Verify** at
> contract time.

### 4.1 Recommendation: **none (Phase 1) → OpenAI gpt-4o-mini-tts
(Phase 2) → ElevenLabs (Phase 3 / T3.3)**

Rationale:

1. **Phase 1 = no TTS.** Vertical social autoplays muted; burned-in
   captions cover 95% of the comprehension gap. Shipping silent is
   2× faster and dodges every music/voice IP question (§9.3).
2. **Phase 2 = OpenAI.** $15/M chars × ~600 chars avg script = **$0.009
   per video**. Quality is good enough for ad copy. EN + LATAM-ES
   both ship out of the box.
3. **Phase 3 = ElevenLabs.** Only when we layer in T3.3 (dealer
   voice clone with founder consent). The story "your customers hear
   _your_ voice on every Reel, even at 2am" is a $200-ACV moat. Pay
   the ElevenLabs tax then.

Never ship Google as the headline option — quality on LATAM-Spanish
is the moat, not the floor. Keep it as the "GCP-mandated dealer"
escape hatch only.

## 5. Script generation

### 5.1 Reuse or new prompt?

**Reuse `buildSystemPrompt` machinery, new task prompt.**

- The base founder-voice prompt in `src/lib/ai.ts` (`BASE_PROMPT`) is
  written for **conversational chat replies under 90 words**. It is
  the wrong size and tone for video scripts.
- BUT: the dealer block (`dealerBlock`) and inventory injection
  pattern (`inventoryBlock`) are exactly what we want — same tone
  calibration, same hours/signature/Calendly context, same
  PROMPT_BUDGET_CHARS discipline.

Proposal: extract a **shared composer** that takes
`{dealer, vehicle, task: 'chat'|'listing'|'video_script'}` and stitches
the right base prompt onto the existing dealer+inventory blocks.

```
src/lib/ai/prompts/
  base.chat.ts         (existing BASE_PROMPT moves here)
  base.listing.ts      (T2.2 prompt)
  base.video_script.ts (NEW — T2.4)
  composer.ts          (NEW — shared assembly)
```

This keeps the founder-voice _signal_ (tone, security rules, language
detection) consistent across all 3 generative surfaces while letting
each have its own task shape.

### 5.2 Output JSON contract

Claude returns a single JSON object the renderer consumes:

```json
{
  "language": "en",
  "hook": "Clean one-owner Civic just hit the lot.",
  "captions": [
    { "t": 1.0, "text": "2019 Honda Civic LX" },
    { "t": 4.5, "text": "42,108 miles · Carfax clean" },
    { "t": 8.0, "text": "Auto · Bluetooth · Back-up cam" },
    { "t": 12.0, "text": "$16,400 — financing on the lot" },
    { "t": 18.0, "text": "Tap to DM — we reply in under a minute" }
  ],
  "voiceover_script": "Clean one-owner 2019 Honda Civic LX, 42 thousand miles, Carfax clean. Sixteen-four. Financing on the lot. DM us — we answer fast.",
  "cta_text": "DM us at @{dealer_slug}",
  "intent_for_caption_post": "test_drive_or_dm"
}
```

Strict schema enforced by `parseModelObject` (same parser the chat
pipeline uses). Captions must be ≤ 6 entries, each ≤ 60 chars,
monotonically increasing `t`, last `t` ≤ duration – 2s.

Input (vehicle row):

T2.2 (listing optimizer, shipped v0.3) already generates listing
title + description + photo_order_hint per vehicle. **Video script
generation should opportunistically reuse the most recent accepted
`listing_suggestions` row** as a "stylistic anchor" injected into the
video-script prompt. This keeps the Reel caption consistent with the
Marketplace listing the dealer just posted.

## 6. Storage

### 6.1 Bucket layout (Supabase Storage)

We currently use no Storage buckets (`grep -rn "storage.from" src/`
returns nothing). T2.4 introduces the first:

```
buckets:
  inventory-videos          (private, RLS-scoped)
    {dealer_id}/{video_id}/source.mp4         # canonical render
    {dealer_id}/{video_id}/source.720p.mp4    # downscale fallback
    {dealer_id}/{video_id}/thumb.jpg          # poster frame at t=2.0
    {dealer_id}/{video_id}/caption.txt        # paste-ready social caption
```

Bucket policy: private + RLS — only the owning dealer reads. All
fetches go through a **server-issued signed URL** (15-minute TTL by
default, 24h for the download API surface).

### 6.2 Signed-URL strategy

- **Dashboard preview**: 15-min signed URL minted on each load. Short
  TTL because the dashboard reloads frequently; bounded blast radius
  if a URL leaks.
- **Download API** (`/api/dashboard/videos/[id]/download`): 24h
  signed URL, served as a 302 redirect. Dealer needs time to copy →
  paste into TikTok/Reels on their phone hours later.
- **Public sharing**: not in v1. If a dealer wants to share, they
  upload manually to their socials. Anti-feature for v1 to keep auth
  surface small.

### 6.3 Retention

- **Hot tier (Storage)**: keep 90 days after generation. After 90d
  the row stays in DB but the MP4 is deleted (`storage_deleted_at`
  timestamp recorded). Regeneration is cheap (§8).
- **DB row retention**: indefinite. Cost is rounding error and it
  preserves analytics ("which prompt template generated the best CTR
  6 months ago?").
- **Cleanup cron**: piggyback on the existing `/api/internal/drain-*`
  cron pattern. Add `/api/internal/expire-videos` that
  soft-deletes Storage objects past TTL.

## 7. Trigger model

### 7.1 Three trigger paths (Phase 1 ships only one)

| Trigger                           | When               | UX                                                         | Phase  |
| --------------------------------- | ------------------ | ---------------------------------------------------------- | ------ |
| **Manual "Generate video" button**| User-initiated     | Per-vehicle row in dashboard inventory table               | **P1** |
| **Bulk "Generate for all stale"** | User-initiated     | Button on T2.3 auto-repost tile — "Refresh marketing"      | **P1** |
| **Auto on new_listing event**     | Event-driven       | Fires when CSV/DMS import inserts a vehicle row            | P3     |

### 7.2 Why button-first

- We don't know yet what _good_ looks like. Surfacing a button gives
  the dealer agency to discard junk early. Auto-generation can be
  embarrassing if the photo is dim, the price is wrong, or the
  inventory was a placeholder.
- T2.3 (auto-repost) already nudges dealers to "refresh" stale
  inventory. Putting a "Generate video" button next to the
  "Mark as reposted" CTA on that same tile is a natural surface —
  one habit, two outputs.
- Auto-generation on new_listing (P3) ties into T2.5 (outbound
  re-engagement): when a new vehicle comes in, we both ping warm
  buyers and pre-bake a Reel for the dealer to push.

### 7.3 Idempotency / rate

- One in-flight render per (dealer, vehicle) at a time. Re-clicking
  the button while a render is "queued" → "rendering" is a no-op
  that returns the existing job id.
- Per-dealer monthly cap (configurable; soft-default 100 renders/mo
  to bound runaway cost). Enforced in the generate API; surfaces in
  the response as `{ status: "cap_exceeded", reset_at: ... }`.

## 8. Cost envelope

### 8.1 Target steady state

- 100 dealers
- 30 vehicles each
- 1 video per vehicle per month
- = **3,000 videos/month** = ~100/day

### 8.2 Per-video cost (recommended stack)

| Cost line                                | Amount   | Note                                                            |
| ---------------------------------------- | -------- | --------------------------------------------------------------- |
| Claude script generation (1× call)       | ~$0.005  | ~3K input + ~400 output tokens at Sonnet 4.6 pricing            |
| Remotion + Lambda render (30s 1080p)     | ~$0.004  | Verified order-of-magnitude from Remotion docs                  |
| S3 storage (90d × ~8 MB)                 | ~$0.0002 | $0.023/GB-mo × 0.008 GB × 3                                     |
| S3 + CloudFront egress (1 download)      | ~$0.001  | ~8 MB × $0.09/GB; usually 1-2 downloads/video                   |
| Supabase Storage egress (if not S3)      | ~$0.0   | Supabase free egress under tier `verify`                        |
| **Phase 1 subtotal (silent video)**      | **~$0.011** | Rounded to **~$0.01/video**                                   |
| TTS (Phase 2: OpenAI mini-tts)           | +$0.009  | 600 chars × $15/1M                                              |
| **Phase 2 subtotal (with TTS)**          | **~$0.02/video** | Rounded                                                  |
| TTS (Phase 3: ElevenLabs Pro)            | +$0.144  | 600 chars × $0.24/1K                                            |
| **Phase 3 subtotal (cloned voice)**      | **~$0.16/video** | Premium tier — only with consenting dealer + T3.3        |

### 8.3 Monthly totals at 3,000 videos

| Phase | $/video | $/month   | $/year     |
| ----- | ------- | --------- | ---------- |
| 1     | $0.011  | ~$33      | ~$396      |
| 2     | $0.020  | ~$60      | ~$720      |
| 3     | $0.160  | ~$480     | ~$5,760    |

For comparison if we had picked **Shotstack** for the render layer:
$0.10/render × 3,000 = **$300/mo** for renders alone — a 9× tax over
Remotion. That's the entire annual margin on one dealer at $99/mo.

### 8.4 Sensitivity

The numbers move most with:

1. **Render provider** (10× swing between Lambda and hosted) — biggest lever.
2. **TTS provider** (16× swing between OpenAI and ElevenLabs) — second lever.
3. **Retention** (90d → 30d cuts Storage 3× but Storage is rounding noise).
4. **Video duration** (30s → 60s ~ doubles render + TTS) — keep ≤30s.
5. **Resolution** (1080p → 720p cuts ~40% but quality penalty is real).

Per-dealer-per-month at $99 plan: even Phase 3 cloned-voice at 30
videos/dealer/month = $4.80 in COGS, ~5% of revenue. Comfortable.

## 9. Failure modes & retries

### 9.1 Long renders / hung Lambda

- Lambda hard ceiling 15 min; our budget is 60s. If a render goes
  past 120s, abort and mark job `failed_timeout`. Render is retried
  once with `quality_tier='720p'` (smaller files → faster) before
  surfacing to the user as `failed_permanent`.
- Vercel Next.js handler **never blocks** on render. Generate
  endpoint enqueues + returns `202 { video_id, status: "queued" }`;
  dashboard polls the `/status` endpoint with exponential backoff
  starting at 2s.

### 9.2 Photo / font fetch failures

- **Missing or 404 `photo_url`**: precondition fail. The generate
  endpoint short-circuits with `412 { error: "no_photo" }` and the
  dashboard shows a "Upload a photo to generate" affordance. No
  Lambda invocation — we don't waste $.
- **Photo fetch slow**: download with 8s timeout + 2 retries. On
  permanent failure, fall back to a branded placeholder
  ("{dealer.name} — call for photos"). Marked `degraded=true` so
  dealer sees a banner.
- **Photo wrong aspect / portrait**: ffmpeg / Remotion pads with a
  blurred-photo backdrop (industry-standard treatment).
- **Fonts**: bundled in the Lambda layer at deploy time — never
  fetched at render. If a future template needs a new font, it ships
  via deploy, not over HTTP.

### 9.3 Copyright / IP

The single biggest non-obvious risk on a video product.

- **Music**: ship NO music in v1/v2. Even "royalty-free" libraries
  bring receipts-to-prove headaches with Meta/TikTok rights
  management. Phase 3 only when we contract a specific licensing
  partner (e.g. Epidemic Sound API) with audit logs.
- **TTS voices**: OpenAI's stock voices have explicit commercial
  rights; ElevenLabs Pro has commercial rights. **Always log which
  voice generated which video** for audit. Never use a voice clone
  without `dealers.voice_clone_consent_at` timestamp on file
  (this lines up with T3.3 and the existing TCPA consent pattern).
- **Photos**: photos are the dealer's content. T&C should
  add "you grant LotPilot a license to your photos and inventory
  data for the purpose of generating marketing assets you've
  requested." Founder + legal to review before T2.4 ships (OQ-3).
- **Stock vehicle logos**: do not composite manufacturer logos into
  the output. We render text only ("Honda Civic"), not a Honda
  wordmark.

### 9.4 Retry policy

```
attempt 1: full template, 1080p
attempt 2 (on failure): same template, 720p
attempt 3 (on failure): no Ken-Burns motion, 720p, static frame
permanent fail: surface to user + create a `system_warnings` row
```

Same pattern as the existing T0.4 / Anthropic budget circuit
breaker. Reuse the `system_warnings` table (introduced v0.6 for
Calendly mismatches) so dealers see one consolidated banner.

### 9.5 Cost circuit breaker

Reuse the existing Anthropic budget pattern: a per-dealer monthly
soft cap (default 100 renders) plus a global daily kill switch
(default 1,000 renders). On breach the generate endpoint returns
`429 { error: "cap_exceeded" }`. Logged + alerts via the same
structured logger v0.2 introduced.

## 10. Phased rollout

### Phase 1 — "Manual button + Remotion + no TTS" (S, ~1.5–2 weeks)

Goal: dealer clicks a button, gets a watchable Reel-shaped MP4 with
captions.

- DB: new `inventory_videos` table (§11) + migration `0013_v08_videos.sql`.
- Storage: `inventory-videos` Supabase bucket with RLS.
- API: `POST /api/dashboard/videos/generate`,
  `GET /api/dashboard/videos/[id]/status`,
  `GET /api/dashboard/videos/[id]/download`.
- Renderer: Remotion project under `src/lib/video/remotion/`
  (NEXT BUILD STEP, not this design doc).

## 11. DB schema (proposed)

New table `inventory_videos`. Sketch — `0013_v08_videos.sql` will be
the actual migration when implementation lands; this section is
prescriptive only.

```sql
create table public.inventory_videos (
  id                  uuid primary key default gen_random_uuid(),
  vehicle_id          uuid not null references public.vehicles(id) on delete cascade,
  dealer_id           uuid not null references public.dealers(id) on delete cascade,

  -- pipeline state
  status              text not null default 'queued'
                        check (status in ('queued','rendering','ready','failed','deleted')),
  failure_reason      text check (failure_reason is null or char_length(failure_reason) <= 500),
  failure_attempts    int  not null default 0 check (failure_attempts between 0 and 5),

  -- inputs (what we asked for)
  template_id         text not null default 'standard-9x16.v1',
  language            text not null default 'en' check (language in ('en','es')),
  voice_provider      text check (voice_provider is null
                        or voice_provider in ('none','openai','elevenlabs','google')),
  voice_id            text check (voice_id is null or char_length(voice_id) <= 100),

  -- ai outputs (what Claude returned, snapshotted for replay/audit)
  script_json         jsonb not null,        -- the §5.2 object verbatim
  prompt_version      text not null,         -- e.g. 'video_script.v1'
  source_photo_url    text not null,         -- snapshot in case vehicle photo changes

  -- render outputs
  storage_path        text check (storage_path is null or char_length(storage_path) <= 500),
  thumb_path          text check (thumb_path is null or char_length(thumb_path) <= 500),
  duration_sec        numeric(5,2) check (duration_sec is null or duration_sec between 0 and 120),
  size_bytes          bigint check (size_bytes is null or size_bytes between 0 and 50_000_000),

  -- cost / audit
  render_cost_cents   int check (render_cost_cents is null or render_cost_cents between 0 and 1000),
  tts_cost_cents      int check (tts_cost_cents is null or tts_cost_cents between 0 and 1000),
  llm_cost_cents      int check (llm_cost_cents is null or llm_cost_cents between 0 and 1000),

  -- lifecycle
  generated_by        text not null check (generated_by in ('manual','bulk','auto_new_listing')),
  generated_at        timestamptz,           -- set when status→ready
  storage_deleted_at  timestamptz,           -- set when cleanup cron expires asset

  created_at          timestamptz not null default now(),
  updated_at          timestamptz not null default now()
);

create index inventory_videos_dealer_status_idx
  on public.inventory_videos(dealer_id, status, created_at desc);
create index inventory_videos_vehicle_idx
  on public.inventory_videos(vehicle_id, created_at desc);
create index inventory_videos_cleanup_idx
  on public.inventory_videos(generated_at)
  where status = 'ready' and storage_deleted_at is null;

alter table public.inventory_videos enable row level security;

create policy inventory_videos_owner_all on public.inventory_videos
  for all to authenticated
  using (exists (select 1 from public.dealers d
                 where d.id = inventory_videos.dealer_id
                   and d.owner_user_id = auth.uid()))
  with check (exists (select 1 from public.dealers d
                      where d.id = inventory_videos.dealer_id
                        and d.owner_user_id = auth.uid()));

-- RLS regression test pattern (matches 0006 / 0010 / 0011 / 0012):
-- 0014_test_videos_rls.sql will insert 2-dealer fixtures and assert
-- that dealer_a cannot see dealer_b's rows under auth.uid()=dealer_a.
```

Notes:

- `script_json` snapshot lets us reproduce any output deterministically
  (matches the `previous_title/description` pattern from T2.2).
- `source_photo_url` is snapshotted on render — if the dealer later
  changes the photo, the video stays consistent.
- Cost columns are denormalized for fast monthly cap math (avoids a
  JOIN through messages/ai_calls).
- Status enum mirrors approvable-queue conventions.

## 12. API surface

All routes live under `/api/dashboard/videos/*` (authenticated,
RLS-scoped through the per-request supabase client — same pattern as
`/api/dashboard/compliance/export`).

### 12.1 `POST /api/dashboard/videos/generate`

Request:
```json
{
  "vehicle_id": "uuid",
  "template_id": "standard-9x16.v1",     // optional, defaults to current
  "language": "auto",                    // 'auto' | 'en' | 'es'
  "voice_provider": "none"               // 'none' | 'openai' | 'elevenlabs'
}
```

Response 202:
```json
{ "video_id": "uuid", "status": "queued", "poll_url": "/api/dashboard/videos/<id>/status" }
```

Response 409 (already in flight):
```json
{ "video_id": "<existing>", "status": "rendering" }
```

Response 412 (no photo):
```json
{ "error": "no_photo", "vehicle_id": "uuid" }
```

Response 429 (cap exceeded):
```json
{ "error": "cap_exceeded", "monthly_used": 100, "monthly_cap": 100, "reset_at": "2026-06-01T00:00:00Z" }
```

### 12.2 `GET /api/dashboard/videos/[id]/status`

Response 200:
```json
{
  "video_id": "uuid",
  "status": "ready",
  "duration_sec": 23.5,
  "size_bytes": 7_834_122,
  "thumb_signed_url": "https://...",        // 15-min TTL
  "preview_signed_url": "https://...",      // 15-min TTL
  "caption_text": "Just-in: 2019 Civic LX, 42K mi, $16,400. DM us...",
  "ai_intent_for_caption_post": "test_drive_or_dm",
  "generated_at": "2026-05-15T12:34:00Z",
  "failure_reason": null
}
```

While `rendering`, omit signed URLs; surface progress as an
opaque `estimated_seconds_remaining` (best-effort from Lambda
heartbeat).

### 12.3 `GET /api/dashboard/videos/[id]/download`

Returns 302 → 24h signed URL pointing at `inventory-videos/...mp4`.
Auth is required for the 302 step (RLS-checked); the signed URL
itself is bare and short-lived. This pattern mirrors the v0.6
compliance-export streaming download.

Headers set on the 302:
- `Content-Disposition: attachment; filename="{stock_number}.mp4"`
- `X-LotPilot-Video-Id: <uuid>` (for support diagnostics)

### 12.4 `POST /api/internal/expire-videos` (cron)

Service-role only. Iterates rows where
`status='ready' AND generated_at < now() - interval '90 days' AND storage_deleted_at IS NULL`,
deletes the Storage objects, sets `storage_deleted_at = now()`. Wired
to the same cron schedule pattern as `/api/internal/drain-audit-queue`.

### 12.5 Auth + RLS posture

- Generate / status / download → user-scoped supabase client
  (`createServerClient` with the request's auth cookie), NOT
  service-role. RLS enforces dealer ownership. Same posture as
  T2.7's compliance export.
- Expire-videos cron → service-role + shared-secret header
  (`X-LotPilot-Cron-Secret`), matching the existing internal cron
  pattern.
- Lambda → S3/Storage write happens with a scoped IAM role that can
  only write under the `{dealer_id}/{video_id}/` prefix supplied by
  our orchestrator. Compromised Lambda cannot exfiltrate another
  dealer's videos.

## Appendix A — Cost math worked out

### A.1 Claude script call

- Input prompt: ~3K chars ≈ ~750 input tokens (BASE_PROMPT-style +
  dealer block + vehicle context).
- Output: ~400 tokens (script JSON).
- Sonnet 4.6 pricing (May 2026, `verify`): roughly $3/M input,
  $15/M output.
- Per call: `750/1M × $3 + 400/1M × $15 ≈ $0.0023 + $0.006 ≈ $0.008`.
  Round to **$0.005–$0.01** allowing for prompt caching of the
  static `BASE_PROMPT` block.

### A.2 Remotion + Lambda render

- Remotion docs explicitly cite "pennies for multiple minutes"
  and ~$0.001 for HelloWorld renders. Our composition is heavier
  (1080p, 30s, motion + captions), so estimate **$0.003–$0.01** per
  render at 1080p / 30s with 1–2 GB Lambda memory.

### A.3 Storage / egress

- 8 MB × 90 days × $0.023/GB-mo ≈ **$0.00017** per video.
- 1 download × 8 MB × $0.09/GB ≈ **$0.0007** per video.
- Rounding to **$0.001** total.

### A.4 OpenAI TTS (Phase 2)

- 600-char voiceover script (typical).
- $15 / 1M chars = $0.000015/char.
- 600 chars → **$0.009 per video**.

### A.5 ElevenLabs Multilingual v2 (Phase 3)

- 600-char voiceover script.
- Pro plan overage rate ≈ $0.24 / 1K chars.
- 600 chars → **$0.144 per video**.

## Appendix B — Why not just outsource the whole thing?

Three "buy" alternatives evaluated:

1. **VEED.io / InVideo / Pictory APIs**: not built for transactional
   B2B SaaS. Pricing scales by seats, not renders. Out.
2. **Shotstack / Creatomate**: covered in §3. Right tool for the
   wrong cost curve at our scale.
3. **Synthesia / HeyGen** (avatar-based): great quality but $1+/min
   and the value prop ("AI human reads your inventory ad") is the
   _opposite_ of independent-dealer authenticity. Out.

Build-on-Remotion is the only option that keeps us cost-competitive
AND visually differentiable from the incumbents.
